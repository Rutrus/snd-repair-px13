Subject: [RFC PATCH 0/3] ASoC: tas2783: mitigate intermittent firmware download failures

Hi,

[EXPERIMENTAL — requesting feedback before a formal submission]

On a 2x TAS2783 SoundWire link (Reported-on: ASUS ProArt PX13, AMD ACP70),
sdw_nwrite_no_pm() during async firmware download intermittently returns
-ETIMEDOUT on the second slave (UID 0xb). The first slave is unaffected.

Reliability matrix (7 cold boots, same hardware):

  Boot | UID 0x8 (Left) | UID 0xb (Right)
  -----|----------------|------------------
    1  | OK             | FAIL (-110)
    2  | OK             | OK
    3  | OK             | WARN (race)
    4  | OK             | FAIL (-110)
    5  | OK             | FAIL (-110)
    6  | OK             | OK
    7  | OK             | OK   (with patches 0006+0007)

Before patches: ~50% FW failure rate on second slave.
After patches (boot 7): 0 failures in limited sample (n=1 post-fix boot).

This series adds:
  1/3 — bounded retries on -ETIMEDOUT/-EAGAIN during FW write
  2/3 — wait in hw_params until async FW load completes
  3/3 — reload FW after system sleep (invalidate hw_init on suspend)

Suspend/resume (post-0003, same hardware, kernel 7.0.0-27): 6/6 cycles with
both TAS2783 slaves FW OK (validation matrix boots #30–#35). PM -110 on
resume may still occur; platform recovery (PCI reset) remains in use.

Not yet completed: cold-boot matrix at scale, 44.1/48/96 kHz sweep (see
VALIDATION-TODO.md). Open question: should retries live in the codec driver
or in the SoundWire bus layer?
