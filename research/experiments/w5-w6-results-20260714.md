## Invalid run — timer 1500 ms (2026-07-14 15:43)

Audio: **none**. Do **not** count as FAIL for delay threshold.

Kernel: PipeWire `port_prep` during W6; uid11 `ret=-11` (EAGAIN); uid8 `ret=0`; speaker-test EBUSY.

**EAGAIN** = readiness race ("not now"), not permanent FW failure. Reinforces device-readiness hypothesis; does not contradict W5 / W6@3000.

Full W2 sequence for both amplifiers takes **~2.7 s** from anchor (uid11 ~1.3 s + uid8 ~1.3 s, sequential in log). Wording: "sequence duration", not proven "blocks bus".

```text
08:29:36  W2 force_fw_reinit uid=11
08:29:36  W6 schedule deferred_reinit uid=11 delay_ms=3000
08:29:36  W2 force_fw_reinit uid=8
08:29:36  W6 schedule deferred_reinit uid=8 delay_ms=3000
08:29:38  W6 deferred fw_reinit uid=11        (~2 s after schedule)
08:29:39  W6 deferred fw_reinit uid=11 ret=0
08:29:39  W6 deferred fw_reinit uid=8
08:29:40  W6 deferred fw_reinit uid=8 ret=0
```

Second reinit completes ~3–4 s after resume. `speaker-test` may hit `-EIO` if it starts while reinit runs — wait for both `ret=0`, then replay.

Snapshots: `validation/w6-delay-0ms-20260714-081926/`, `validation/w6-delay-3000ms-20260714-082858/`

## What is now demonstrated

1. **W5 reproducible** — userspace 2nd `fw_reinit()` consistently restores audible playback.
2. **W6 @ 3000 ms PASS** — automated 2nd reinit works; @ 0 ms FAIL (control).
3. **Race hypothesis** — same code path, different **when**; not FW/init_seq corruption.
4. **1500 ms** — pending (one test closes timing vs context).
5. **W8** — hw_params / port_prep / dapm POST_PMU triggers at **0 ms** distinguish pipeline milestone from sleep.

Still open: exclusive proof of "too early" vs "wrong context" — see [w8-context-reinit-protocol.md](w8-context-reinit-protocol.md).
